## 라우팅의 핵심 발상

> "이 우편물 어디로 보내야 하지?" 를 *목적지 주소*만 보고 결정.

각 컴퓨터·라우터는 **라우트 테이블**을 들고 있고, 패킷의 목적지 IP를 보고 "이 IP면 저쪽 게이트웨이로 가" 하고 던짐. 받은 다음 게이트웨이가 또 자기 테이블로 결정. **이 과정이 반복**해서 결국 목적지에 도착.

→ "전국 우편 배송"과 똑같음. 우체국 직원이 *최종 주소까지* 다 아는 게 아니라, "이건 부산행 트럭" 하고 분류만 하면, 부산 우체국이 또 분류함.

## 라우트 테이블 — 분류표

> 한 줄 = "이 목적지 IP 묶음으로 가려면 이 경로로 가라"

### 예시 (AWS public subnet의 라우트 테이블)
```
목적지(Destination)  →  타깃(Target)
─────────────────────────────────────
10.0.0.0/16          →  local             ← "같은 VPC면 그냥 안에서 처리"
0.0.0.0/0            →  igw-xxxxxxxx      ← "그 외 모두(=인터넷)는 이 IGW로"
```

→ 패킷이 도착하면 **가장 *구체적인* 룰부터 매칭** (longest prefix match):
- 목적지 `10.0.1.5` → `10.0.0.0/16`이 매칭(16비트 일치) → `local`로 = VPC 내부 라우팅
- 목적지 `8.8.8.8` → `10.0.0.0/16` 매칭 안 됨 → `0.0.0.0/0`이 매칭 → IGW로

### "0.0.0.0/0" = "그 외 전부" = default route
- 모든 IP가 매칭됨 (32비트 중 0비트 일치).
- 가장 마지막 폴백.
- "이 룰이 IGW면 퍼블릭, NAT GW면 프라이빗" — 서브넷의 *공개성*을 결정.

### "local"이 의미하는 것
- AWS가 자동으로 추가하는 룰. VPC CIDR 안의 IP는 *내부 라우팅*으로 처리.
- 삭제 불가.
- → 같은 VPC의 서브넷끼리는 *기본적으로 통신 가능* (SG로 추가 차단 가능).

### 우리 프로젝트 — Ch 10 Phase A
```
public RT:
  10.0.0.0/16 → local
  0.0.0.0/0   → IGW           ← 퍼블릭 서브넷이 인터넷 보고/들리는 길

private RT:
  10.0.0.0/16 → local
  0.0.0.0/0   → NAT GW        ← 프라이빗 서브넷이 인터넷 *보내기*만 가능
```

→ "퍼블릭/프라이빗"의 본질 = **`0.0.0.0/0`이 IGW냐 NAT냐**.

### 함정
- 라우트 테이블이 **서브넷에 *연결(associate)*되어 있어야** 효력. 서브넷 만들고 연결 안 하면 default RT 사용 (보통 의도와 다름).
- 라우트가 *겹치면* longest prefix match로 결정.
- 라우팅은 *단방향*만 정의. 양쪽 다 라우팅이 맞아야 통신 가능.

## Internet Gateway (IGW) — 단지 정문

> 단지와 인터넷을 잇는 **양방향 통로**. VPC당 1개.

### 역할
- VPC 안 → 인터넷: 패킷에 *공인 IP*가 있으면 인터넷으로 내보냄.
- 인터넷 → VPC 안: 공인 IP를 가진 서브넷의 인스턴스로 들여보냄.

### 작동 조건 — 4가지 동시에 만족해야 인터넷 통함
1. VPC에 IGW가 **attach**되어 있음
2. 라우트 테이블에 `0.0.0.0/0 → IGW` 룰
3. 그 라우트 테이블이 서브넷에 **연결**됨
4. 인스턴스에 **공인 IP**가 있음 (서브넷의 `map_public_ip_on_launch=true` 또는 Elastic IP 부여)

→ **하나라도 빠지면 안 됨.** Ch 08-B에서 "퍼블릭 서브넷 4요소"로 학습한 것. EC2 SSH가 타임아웃이면 이 4개부터 점검.

### 비용
- **무료** (IGW 자체). 데이터 전송 비용은 별개.

### 함정
- IGW가 있다고 *자동으로 인스턴스가 인터넷됨* X. 위 4 조건 다 필요.
- VPC 1개당 IGW 1개만 가능.

## NAT Gateway — 정문 경비실 우편함

> **프라이빗 서브넷**의 인스턴스가 *외부로 우편 보낼 때만* 거치는 경비실.

### 역할 (한 줄)
프라이빗 서브넷에서 outbound 인터넷 요청을 받아, *발신자 IP를 NAT의 공인 IP로 바꿔* 인터넷으로 내보냄. 응답이 돌아오면 원래 인스턴스로 다시 전달.

### 비유 — 우편 대행
```
1. 1505호 (private subnet의 EC2): "이 우편 외부로 보내주세요"
2. 경비실 (NAT GW): 발신자를 "경비실 주소"로 바꿔서 우편 부침
3. 외부에서 응답: "경비실 주소" 앞으로 답장 도착
4. 경비실: "아 이거 1505호 우편이었지" 기억하고 1505호로 전달
```

→ 그래서 **외부에서 1505호로 *먼저* 우편 보낼 수 없음**. 경비실은 누구한테 갈지 모름.

### 어디 두나
- **퍼블릭 서브넷**에 둠 (NAT GW 자체가 인터넷이랑 통신해야 하니까).
- Elastic IP 1개 필요 (NAT의 공인 IP).
- AZ당 1개 둘 수 있음 (HA용). 우리 Ch 10은 단일 NAT (ADR-003).

### 작동 조건
1. NAT GW가 퍼블릭 서브넷에 있음 + EIP 할당
2. 프라이빗 서브넷의 라우트 테이블에 `0.0.0.0/0 → NAT`
3. 라우트 테이블이 프라이빗 서브넷에 연결됨
4. (선택) NAT GW가 있는 *퍼블릭* 서브넷의 라우트도 정상 (IGW로)

### 비용 (작지 않음)
- **시간당 $0.045** (~$32/월 단일)
- + 데이터 처리 GB당 $0.045
- 24/7 운영하면 NAT가 학습 프로젝트 비용의 1위가 됨

→ "destroy 안 하고 자면 하루 $1+ 새어나가는 자원". 우리 캡스톤이 매 세션 destroy 하는 큰 이유.

### "왜 안 쓰면 안 되나?"
- 프라이빗 EC2에 패치(`apt update`) / 컨테이너 이미지 pull / Secrets Manager API 호출 등이 다 outbound 필요.
- "프라이빗인데 외부 못 함" = NAT 빠뜨림. 가장 흔한 함정.

### 함정
- NAT GW는 **생성·삭제 모두 ~2분** (가장 느린 자원). apply 중 "Still creating" 메시지 정상.
- destroy 시 NAT 먼저 지워야 EIP 해제됨 (Terraform이 자동 처리).
- EIP가 *NAT에 안 붙어있으면* (분리 상태) 시간당 ~$0.005 과금. 정리 잘 안 하면 새어나감.

## IGW vs NAT GW 비교

가장 자주 헷갈리는 두 개 정확히:

| | **IGW** | **NAT GW** |
|---|---|---|
| **위치** | VPC에 attach | 퍼블릭 서브넷 안에 |
| **방향** | 양방향 (내→외, 외→내) | **outbound만** (내→외, 응답만 외→내) |
| **외부 접속** | 가능 (인스턴스가 공인 IP 있으면) | 불가 (외부에서 먼저 시작 X) |
| **EIP** | 불필요 | **필요** (1개) |
| **개수** | VPC당 1개 | AZ당 1개 가능 |
| **비용** | 무료 | $0.045/hr + 데이터 |
| **사용 대상** | 퍼블릭 서브넷 | 프라이빗 서브넷 |

### "왜 둘 다 필요?"
- IGW만 있으면 → 인스턴스가 *공인 IP 가져야* 인터넷됨 → 모든 인스턴스에 공인 IP 부여 → 외부 공격 표면 ↑
- NAT만 있으면 → 외부에서 들어올 수 없음 (웹 서버 못 함)
- 둘 다 → 웹 서버는 퍼블릭 + DB/내부 API는 프라이빗. **공격면 ↓, 비용 적절**.

### 비유
- IGW = 단지 **정문**. 누구나 들어올 수도, 나갈 수도.
- NAT GW = 정문 옆 **경비실 우편함**. 안쪽 사람이 우편 부치는 *대행 서비스*. 외부에서 안쪽으로 *먼저* 우편 못 부침.

## 패킷 흐름 예시

진짜 패킷이 어떻게 이동하는지 그림으로:

### 1. 퍼블릭 서브넷의 EC2가 google.com에 GET 요청
```
EC2 (10.0.1.10, public IP 52.79.X.X)
   │ 1. DNS 조회: google.com → 142.250.X.X
   │
   │ 2. 패킷 만듦:
   │    src: 52.79.X.X    dst: 142.250.X.X    port: 443
   │
   ▼ (라우트 테이블 조회)
public RT:
   10.0.0.0/16 → local      ← 매칭 X
   0.0.0.0/0   → IGW         ← 매칭 ✓
   │
   ▼
IGW (단지 정문)
   │ 3. 인터넷으로 전송 (src는 그대로 52.79.X.X)
   │
   ▼
🌐 인터넷 → google
   │
   ▼ (응답)
google → 52.79.X.X 응답
   │
   ▼
IGW → EC2 → 받음
```

### 2. 프라이빗 서브넷의 EC2가 apt update 시도
```
EC2 (10.0.11.10, no public IP)
   │ 1. DNS 조회: archive.ubuntu.com → 91.X.X.X
   │
   │ 2. 패킷 만듦:
   │    src: 10.0.11.10   dst: 91.X.X.X    port: 80
   │
   ▼ (라우트 테이블 조회)
private RT:
   10.0.0.0/16 → local      ← 매칭 X
   0.0.0.0/0   → NAT GW      ← 매칭 ✓
   │
   ▼
NAT GW (퍼블릭 서브넷에 있음, EIP 52.79.60.38)
   │ 3. ⭐ 발신자 변환 (NAT):
   │    src: 10.0.11.10  →  src: 52.79.60.38
   │    dst: 91.X.X.X    (그대로)
   │    + "10.0.11.10에 이 응답 돌려줘야 함" 메모
   │
   ▼
public RT (NAT GW가 사는 서브넷의):
   0.0.0.0/0 → IGW
   │
   ▼
IGW → 인터넷 → archive.ubuntu.com
   │
   ▼ (응답)
archive → 52.79.60.38 (NAT의 EIP) 응답
   │
   ▼
IGW → public RT(local) → NAT GW
   │
   │ 4. ⭐ 역변환 (메모 보고):
   │    dst: 52.79.60.38  →  dst: 10.0.11.10
   │
   ▼
private subnet → EC2 → 받음
```

→ NAT의 *메모*(connection tracking table) 덕에 응답이 정확한 인스턴스로. 외부에서 *먼저* 보낸 우편엔 메모가 없어서 어디로 갈지 모름 → 차단.

## 치트시트

### "라우트 테이블 보면 알 수 있는 것"
- `0.0.0.0/0 → IGW` 있으면 → 퍼블릭 서브넷
- `0.0.0.0/0 → NAT GW` 있으면 → 프라이빗 서브넷
- `0.0.0.0/0` 없으면 → 외부 인터넷 안 됨 (isolated subnet)

### "이게 안 되면 의심해야 할 것"
| 증상 | 의심 |
|------|------|
| 퍼블릭 EC2 SSH 안 됨 | 4요소: IGW attach? RT에 `0.0.0.0/0→IGW`? RT 서브넷 연결? 공인 IP? + SG 22 허용? |
| 프라이빗 EC2 apt update 안 됨 | NAT GW 있나? private RT에 `0.0.0.0/0→NAT`? RT 서브넷 연결? NAT GW가 *퍼블릭* 서브넷에? |
| VPC 안 인스턴스끼리 통신 X | SG가 *같은 VPC 트래픽* 허용? (NACL 검토는 그 다음) |

### AWS CLI 디버깅
```bash
# 라우트 테이블 확인
aws ec2 describe-route-tables --filters Name=vpc-id,Values=<vpc-id>

# 서브넷-RT 연결 확인
aws ec2 describe-route-tables --query 'RouteTables[].{Id:RouteTableId,Subnets:Associations[].SubnetId,Routes:Routes[].{Dest:DestinationCidrBlock,Target:GatewayId||NatGatewayId}}'

# NAT GW 상태
aws ec2 describe-nat-gateways --filter Name=vpc-id,Values=<vpc-id> --query 'NatGateways[].{Id:NatGatewayId,State:State,EIP:NatGatewayAddresses[0].PublicIp}'
```

### 비용 의식
- IGW: **무료**
- NAT GW: **$0.045/hr + 데이터 GB당 $0.045** — 학습 자원 중 1위
- EIP attached to running: 무료
- EIP unattached: **시간당 $0.005** — 떨어진 EIP 정리 안 하면 새어나감

### 한 줄 정리
> **IGW는 양방향 정문(무료), NAT는 outbound 우편대행($0.045/hr).**
> **퍼블릭/프라이빗의 본질 = `0.0.0.0/0`이 어디로 가느냐.**
